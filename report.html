GF(2m)

Polynomial Arithmetic Calculator

EECE 455/632 — Cryptography & Network Security

Project Report • March 2026

1. Introduction

This report presents the GF(2m) Polynomial Arithmetic Calculator, a web-based tool built for EECE 455/632. It allows users to perform and visualize polynomial operations within Galois Fields of characteristic 2, which are fundamental to cryptographic systems like AES and elliptic curve cryptography. The application is a client-side Single Page Application using pure HTML, CSS, and JavaScript with zero external dependencies — it runs by simply opening index.html in any modern browser.

2. Features Implemented

The calculator implements six GF(2m) operations:

OperationSymbolAlgorithm
Addition⊕Bitwise XOR of two polynomials
Subtraction⊖Same as addition (in GF(2), −1 = 1)
Multiplication×Shift-and-XOR + modular reduction
Division÷A × B−1 using Extended Euclidean Algorithm
Mod Reduce%Iterative XOR reduction to degree < m
Inversex−1Extended Euclidean Algorithm with verification

Additional features include:

  • Configurable field degree (m = 2 to 128) with quick-select buttons for m = 2–8 and a custom input field
  • Three I/O formats: Binary, Hexadecimal, and Polynomial notation — values auto-convert when switching formats
  • Irreducible polynomial management: 15 pre-loaded defaults (m = 2–16), editable input with real-time degree validation, and a reset button
  • Step-by-step breakdown: Every operation generates a detailed trace of each intermediate computation step
  • Swap button to exchange inputs A and B; copy-to-input buttons to feed results back for chained operations
  • Live preview below each input showing the polynomial (or binary/hex) equivalent as the user types

3. Architecture & How It Works

3.1 Project Structure

The entire application is three files (~1,572 lines total):

  • index.html (106 lines) — Page structure and layout
  • style.css (671 lines) — Theming, responsive layout, animations
  • gf_arithmetic.js (795 lines) — Math engine, parser, and UI controller

There is no backend, database, or external API. All computation happens client-side in the browser.

3.2 Polynomial Representation

Polynomials are represented as JavaScript BigInt bit-vectors, where each bit position corresponds to a coefficient. For example, x4 + x2 + 1 = 10101 in binary = 21n as a BigInt. Using BigInt allows support for degrees up to 128 without precision issues.

3.3 Data Flow

User types input → Format-aware parser converts string to BigInt → Validation (degree, format, irreducible poly) → Operation executes while recording steps → Result rendered in all formats + step timeline displayed

3.4 State Management

A singleton App object holds four state values: the field degree m, the current I/O format, the irreducible polynomial, and the last result. All state changes propagate immediately to the DOM — changing degree reloads the default irreducible polynomial and clears results; switching format re-displays existing values.

3.5 Input Parsing

The parser accepts flexible polynomial notation: x^4 + x^2 + 1, x4+x2+1, 3x^3 + 2x + 1 (coefficients auto-reduced mod 2), and even pasted Unicode superscripts like x⁴. Subtraction signs are treated as addition (since −1 = 1 in GF(2)). Binary and hex inputs are validated with strict character checks.

4. UI Design

The interface uses a dark-mode theme with a blue accent color palette, defined through 20+ CSS custom properties for consistency. The layout is organized vertically: header, configuration bar, irreducible polynomial section, two input cards with a swap button, operation buttons in a grid, result display, and step-by-step timeline.

Key design choices:

  • Typography: System sans-serif for UI, monospace (Cascadia Code/Consolas) for binary/hex values, and math serif (Cambria Math) for polynomial notation
  • Responsive: Single breakpoint at 640px — inputs and config stack vertically on mobile, operation grid switches from 4 to 3 columns
  • Visual feedback: Slide-in animations for results, hover lift on buttons, blue glow focus rings, color-coded step labels (yellow = section headers, green = final result, red = errors)
  • Scrollable step timeline with a custom thin scrollbar for long computation traces

5. Validation & Error Handling

The application validates at multiple levels: format-specific input characters, polynomial degree vs. field degree (warns if degree ≥ m), irreducible polynomial degree match, division/inverse of zero detection, and GCD checks during inverse computation. Errors appear contextually — below the relevant input field or in the result section with red styling.
